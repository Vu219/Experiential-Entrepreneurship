# Business_Analysis.md — AI-Powered Automatic Content Posting (AIMA)

> Platform scope: **Facebook → Instagram → Threads** (rollout in that order — see REQUIREMENTS.md). TikTok, YouTube Shorts, and LinkedIn are planned for later phases and are out of the current scope.

The main actors are:

1. Admin  
   **Role**: Manages the system, monitors errors, and handles content rejected by the platforms.

2. User / Business Owner  
   **Role:** Uses the tool and provides its inputs (brand identity, content strategy, and posting schedule). The user is ultimately responsible for the content published to their accounts (AS-04).

3. Agent AI  
   **Role:** Automatically researches trends, generates content, formats posts, analyzes performance, and publishes posts automatically.

4. Social Platforms  
   **Platforms:** Facebook, Instagram, Threads (current scope)

5. Analytics System  
   **Role:** Collects performance metrics.

Business flow:

Main flow: Automated Content Operations  
The goal is for the AI to autonomously create, schedule, publish, and optimize content across multiple social media platforms.

The steps are:

1. The customer creates a brand profile.  
2. The customer defines the content strategy they want to pursue.  
3. The system records and saves the configuration.  
4. The Agent AI automatically generates relevant content ideas.  
5. The Agent AI creates scripts, captions, hashtags, and media prompts (text descriptions of the image/video to use — the MVP does not generate media itself, FR-29).  
6. The Agent AI formats the content appropriately for each platform.  
7. The Agent AI schedules posts for golden hours (the time slots with the highest engagement on each platform).  
8. The system publishes posts automatically.  
9. The Agent AI researches industry trends to refresh ideas for upcoming cycles.  
10. The system collects performance data generated by each post.  
11. The Agent AI analyzes the results.  
12. Based on the analysis results and researched trends, the Agent AI proposes strategy adjustments; the user accepts or rejects them (FR-68).

# **Detailed Business Flows**

## **Flow 1: Brand Persona Setup (BF01)**

**Flow name:** Brand Persona Setup

**Actor:** User

**Description:**  
 The user performs a one-time configuration of the foundational information the AI needs to understand the brand.

**Inputs include:**

| Information | Examples |
| ----- | ----- |
| Industry | Cosmetics, education, fashion, F\&B |
| Brand voice | Professional, humorous, youthful, luxurious |
| Target audience | Gen Z, office workers, students, shop owners |
| Content goals | Increase brand awareness, increase orders, drive traffic |
| Platforms used | Facebook, Instagram, Threads |
| Posting frequency | 2 posts/day, 5 posts/week |
| Preferred time slots | 8 AM, 12 PM, 8 PM |

**Process:**

1. The user opens the Agent AI configuration page.  
2. The user enters the brand information.  
3. The user selects the platforms to post on.  
4. The user sets the posting frequency and schedule.  
5. The system validates the information (name, industry, and audience required; at least one platform; valid frequency — FR-09).  
6. The system saves the configuration.  
7. The Agent AI creates an internal brand profile.

**Outcome:**  
 The AI understands the "brand voice" and applies it to all future content.

---

## **Flow 2: Content Generation (BF03)**

**Flow name:** Content Generation Workflow

**Actor:** Agent AI

**Goal:**  
 The AI produces complete content from the brand profile, strategy, and trend/idea inputs.

**Process:**

1. The AI receives the brand profile, the active strategy, and the list of trends/ideas.  
2. The AI selects one or more ideas.  
3. The AI writes a video script (hook, main content, shot suggestions, CTA — FR-25).  
4. The AI writes a caption.  
5. The AI generates hashtags.  
6. The AI writes a media prompt — a text description of the image/video to use (the MVP does not generate media itself, FR-29).  
7. The AI checks that the content matches the brand voice.  
8. The AI saves the content as a draft (`Draft`/`Generated`). The user can review, edit, or regenerate it before it is scheduled (FR-32 to FR-34).

**Output may include:**

| Component | Content |
| ----- | ----- |
| Script | Short-form video script |
| Caption | Post description text |
| Hashtag | Platform-specific hashtag set |
| Media prompt | Text description of the image/video to use |
| CTA | Call to action |

---

## **Flow 3: Platform-Specific Formatting (BF04)**

**Flow name:** Platform Formatting Workflow

**Actor:** Agent AI

**Goal:**  
 The AI automatically adapts content to each platform's conventions, producing one ContentVersion per platform.

**Examples (current scope):**

| Platform | How the AI handles it |
| ----- | ----- |
| Facebook | Longer caption, clear CTA, shareable, combines image/video/link |
| Instagram | Vertical video or square/vertical image, emotive caption, brand hashtags, highly visual |
| Threads | Short, conversational format; details follow the Threads API capabilities |

*(TikTok, YouTube Shorts, and LinkedIn formatting are out of the current scope.)*

**Process:**

1. The AI checks the list of platforms the user has connected.  
2. The AI retrieves the original content (ContentItem).  
3. The AI adjusts the media specification.  
4. The AI rewrites the caption for each platform.  
5. The AI optimizes the hashtags.  
6. The AI creates a dedicated ContentVersion for each channel.  
7. The system saves the formatted versions.

**Outcome:**  
 A single content idea can be turned into multiple distinct posts, one per platform.

---

## **Flow 4: Scheduling and Auto-Posting (BF05)**

**Flow name:** Schedule & Auto Posting Workflow

**Actors:** Agent AI, Social Platforms

**Goal:**  
 The AI publishes posts automatically at the optimal time.

**Process:**

1. The AI checks the configured posting schedule.  
2. The AI selects a suitable post.  
3. The AI determines the golden hour for posting (defaults: Facebook 8–9 AM, 1–2 PM, 8–9 PM; Instagram 8–10 AM, 12–1 PM, 7–9 PM; switches to data-driven suggestions after ≥10 posts with analytics — FR-48).  
4. The system places the post in the publishing queue.  
5. When the scheduled time arrives, the system calls the platform's API (asynchronously, as a background job).  
6. The platform returns a success or failure result.  
7. The system saves the post status.  
8. If the post succeeds, the system begins tracking its performance.

**Post statuses (see the full state machine below):**

| Status | Meaning |
| ----- | ----- |
| Draft | Draft content |
| Generated | Created by the AI |
| Need Review | Awaiting user review |
| Approved | Approved by the user |
| Formatted | Formatted per platform |
| Scheduled | Scheduled for posting |
| Posting | Being published |
| Posted | Published successfully |
| Failed | Publishing failed |
| Retrying | Being retried |
| On Hold | Waiting for the user to reconnect an expired account |
| Analyzing | Performance being analyzed |
| Optimized | Used to optimize the strategy |

---

## **Flow 5: Trend Research (BF06)**

**Flow name:** Trend Research Workflow

**Actor:** Agent AI

**Goal:**  
 The AI proactively discovers trends relevant to the user's industry to feed the next content cycles.

**Process:**

1. The research runs automatically on schedule (default **2:00 AM daily**) or when the user presses "Research now" (FR-19). It only runs if a Brand Profile exists and a Strategy is `Active`, and a new session does not start while a previous one is still running.  
2. The AI collects trend data from the platforms.  
3. The AI filters trends by the user's industry (FR-20).  
4. The AI rates each trend's fit with the brand: High / Medium / Low (FR-21).  
5. The AI selects the most promising trends.  
6. The AI produces a list of content ideas based on those trends and saves the research session (FR-22, FR-23).

**Example output:**

| Trend | Platform | Relevance | Idea |
| ----- | ----- | ----- | ----- |
| "A day in the life of…" | Instagram | High | A day in the life of an online shop owner |
| Before/After | Instagram | High | Before and after using the product |
| Quick tips | Threads | Medium | 5 tips to boost work productivity |

**Outcome:**  
 The system has a fresh list of content ideas tailored to each platform.

---

## **Flow 6: Post Performance Analysis (BF07)**

**Flow name:** Post Performance Analysis Workflow

**Actors:** Analytics System, Agent AI

**Goal:**  
 The AI learns from post results to improve future posts.

**Metrics analyzed:**

| Metric | Meaning |
| ----- | ----- |
| Views | Number of views |
| Likes | Number of likes |
| Comments | Number of comments |
| Shares | Number of shares |
| Saves | Number of saves |
| CTR | Click-through rate |
| Conversion | Conversion rate |
| Watch time | Video watch duration |

**Process:**

1. After a post is published, the system waits for a set period (metrics are collected at 24h, 48h, and 7d; analytics may not be real-time — CON-04).  
2. The system pulls engagement data from the platform.  
3. The AI compares performance across posts.  
4. The AI identifies which posts performed well.  
5. The AI analyzes the success factors: hook, caption, hashtags, CTA, media, posting time, platform (FR-63).  
6. The AI generates optimization insights (FR-64).  
7. The AI proposes strategy adjustments; if the user accepts, the strategy is updated and applied to the next content cycle (BF08, FR-68).

**Outcome:**  
 Future content steadily improves because the AI learns from real data.

# **Key Exception Flows**

## **Exception 1: Social media account not connected (EX-01)**

**Situation:**  
 The user has not connected Facebook/Instagram/Threads.

**Handling:**

1. The system checks whether the account is connected before posting (FR-17).  
2. If not, the system blocks posting and reports the issue (FR-69).  
3. The user is prompted to connect the account.  
4. Once connected, the system resumes the posting process.

---

## **Exception 2: Content violates platform policy (EX-02)**

**Situation:**  
 A platform rejects the content for a policy violation during upload, or flags it after publication.

> Per SEC-06, the system does **not** build its own content filter. Content safety relies on the user review step before scheduling (`Need Review` → `Approved`) and on handling the platforms' violation responses.

**Handling:**

1. The platform returns a policy violation error (HTTP 400/403) during publishing, or sends a violation notification after publication.  
2. The system classifies the error as a policy violation (distinct from technical errors — FR-37), moves the post to `Failed`, and stores the original error code and message. **No retry** is attempted for this error type.  
3. The system notifies the user: which platform, the reason, and the next steps (FR-38).  
4. The user edits the content or asks the AI to regenerate a safer version.  
5. After the revised content is approved, it can be scheduled again (FR-39).

---

## **Exception 3: Publishing failure (EX-03)**

**Possible causes:**

| Cause | Example |
| ----- | ----- |
| Expired token | Facebook token expired |
| API error | Platform API not responding |
| Invalid media file | Video too long, image wrong dimensions |
| Restricted account | Platform blocks automated posting |

**Handling:**

1. The system logs the error and moves the post to `Failed`, storing the error details.  
2. If the error is temporary (timeout, rate limit, network), the system retries up to 3 times at 5/15/30-minute intervals (FR-56).  
3. If the token has fully expired, the account is marked `Expired` and that platform's `Scheduled` posts move to `On Hold` until the user reconnects (FR-18b).  
4. If the system cannot resolve the error itself, it notifies the user.  
5. The user can edit the content/media/time, reconnect the account, or repost (FR-58).

# **Post State Machine**

A post can move through the following states:

Draft  
  ↓  
Generated  
  ↓  
Formatted  
  ↓  
Scheduled  
  ↓  
Posting  
  ↓  
Posted  
  ↓  
Analyzing  
  ↓  
Optimized

Failure path:  
Posting  
  ↓  
Failed  
  ↓  
Retrying  
  ↓  
Posted

Review path:  
Generated  
  ↓  
Need Review  
  ↓  
Approved  
  ↓  
Scheduled

Token-expiry path:  
Scheduled  
  ↓  
On Hold (waiting for the user to reconnect)  
  ↓  
Scheduled

**Summary of post states for the documentation:**
Normal: Draft → Generated → Formatted → Scheduled → Posting → Posted → Analyzing → Optimized

Failure path: Posting → Failed → Retrying → Posted

Review path: Generated → Need Review → Approved → Scheduled

Token-expiry path: Scheduled → On Hold → (reconnect) → Scheduled
